# 维护机制

CodexWorkspace 的价值不在于一次性创建文件，而在于持续维护。

## 需要更新控制台的情况

- 工作区规则变化。
- 新增全局文档。
- 文件职责发生变化。
- 用户改变长期工作方式。

## 需要更新项目台账的情况

- 新建正式项目。
- 外部项目迁移进工作区。
- 项目暂停、恢复、归档。
- 项目路径、入口或使用方式变化。

## 需要更新自动化台账的情况

- 新建定时自动化、提醒、巡检或外部投递任务。
- 自动化的运行时间、时区、读取范围、输出位置或接收方变化。
- 自动化暂停、恢复或停用。
- 自动化从“只输出报告”改为“允许写入工作区文件”。
- 自动化完成真实链路验证或客户端展示验证。

## 需要更新问题日志的情况

- Codex 的结果让用户不满意。
- 处理过程出现明显错误。
- 因规则不清造成返工。
- 发现了值得下次复用的防错规则。
- 自动化或外部通知出现真实投递失败、客户端展示异常、披露风险拦截或附件/权限问题。

## 需要更新 skills 清单的情况

- 新增 skill。
- 删除 skill。
- 重命名 skill。
- 调整 skill 来源、用途或分组。
- 个性化修改或替换已有 skill。
- skill 安装在工作区外，但会影响当前 Codex 全局可用能力。

skill 安装目录只代表本机生效位置，不等于真实来源；清单中应优先记录 GitHub 仓库、插件名称、系统来源或用户说明。

## 维护原则

- 只记录有复用价值的信息。
- 不记录无意义流水账。
- 变化发生在哪里，就更新对应文件。
- 总控文件只保留入口，不承载所有细节。
- 外部投递类问题要记录“配置是否成功、真实链路是否成功、客户端实际展示是否成功”三层结论。
