# 自动化与外部投递

自动化的价值不只是“定时运行”，而是把可复用的工作流稳定地交给 Codex 执行。凡是自动化会读取本地工作区文件，并把结果发到飞书、邮件、GitHub、Slack、网页或其他外部系统，都应同时处理三件事：任务边界、信息披露边界、真实投递效果。

## 适用场景

- 定时读取 `codex-issue-log.md`，整理相似错误和防复发待办。
- 定期生成项目周报、风险报告、维护清单。
- 监控外部状态后，把摘要发送到协作工具。
- 在当前线程之外长期运行的提醒、巡检或复盘任务。

## 典型案例：每周整理 codex-issue-log 并投递到飞书

这是 CodexWorkspace 最值得自动化的治理任务之一：`codex-issue-log.md` 会持续记录错误、不满意、返工和踩坑，但如果不定期归纳，它会逐渐变成日志堆，而不是防复发系统。

推荐自动化目标：

- 每周固定时间读取 `codex-issue-log.md` 和 `AGENTS.md`。
- 识别相似错误、重复复发模式和高频风险。
- 检查 `codex-issue-log.md` 中是否存在重复复盘条目，将同类问题整理成一组。
- 对已经解决但重复出现的条目，输出“建议合并到哪一条、保留哪些证据、如何标注状态”的整理建议。
- 对照 `AGENTS.md` 判断已有规则是否覆盖这些问题。
- 输出“重复问题组、建议合并条目、根因、已有防线、缺失防线、下周待办”。
- 不在每轮最终回复前强制检查旧问题；重复复盘只由定时自动化集中处理。
- 不自动修改治理文件，除非用户明确授权合并或改写 `codex-issue-log.md`。
- 将完整报告写入 `temp/codex-error-audit-report-YYYY-MM-DD.md`。
- 向飞书或其他协作工具发送一条短摘要，并附上完整报告文件。

示例输出结构：

```text
Codex 错误复盘审计报告

1. 本周最高风险复发模式
2. 相似错误分组
3. 重复复盘合并建议
4. 已有规则覆盖情况
5. 缺失防线和建议待办
6. 下周自动化重点观察
```

示例飞书投递方式：

```text
Codex 周报｜本周发现 3 类复发风险：skill 清单遗漏、外部通道验证不足、Windows 命令预检缺失。完整报告见附件。
```

飞书、邮件、Slack 等聊天工具不适合承载完整长报告。即使 API 返回成功，也可能出现 Markdown 卡片只显示标题、多行文本只显示第一行、长文本折叠等问题。完整报告应优先作为 `.md` 或 `.txt` 附件发送。

## 创建前检查

创建自动化前，先确认以下问题：

- 读取哪些文件或数据源。
- 运行频率、时区和触发时间。
- 输出是仅在 Codex 中展示，还是发送到外部系统。
- 外部接收方是谁，是否会被群成员或第三方系统看到。
- 自动化是否允许修改工作区文件，还是只生成报告和建议。
- 失败时如何反馈：静默失败、在 Codex 中报告，还是发送失败摘要。
- 是否需要登记到 `automation-index.md`，以及由谁维护状态、运行时间和投递目标。

## 自动化台账

如果工作区中维护 `automation-index.md`，创建、更新、暂停、恢复或删除自动化后都应同步登记。

台账应同时照顾人和 Codex：

- 人读总览：用一张短表展示编号、场景、状态、触发方式、运行时间、面向人的产出、外部投递和最后验证时间。
- Codex 读取详情：每个自动化编号下面都要有独立场景详情，避免把所有字段挤进一张横向长表。

每个场景详情至少记录：

- 自动化名称、类型和状态。
- 运行时间、时区和工作目录。
- 输入文件或数据源。
- 输出文件位置。
- 外部投递目标和投递方式。
- 自动化是否允许写入工作区文件。
- 失败处理方式和最后验证时间。
- 面向人的成功标准、暂停条件、展示验证方式和维护备注。

外部投递目标应使用占位或简写，不要把完整 token、App Secret、邮箱密码、GitHub 个人访问令牌等凭据写入台账。

## 外部投递三步法

### 1. 先做最小真实链路验证

不要只看配置文件或 masked secret。外部通道必须用最小内容做一次真实发送或真实调用。

示例：

- 飞书 Bot：先发一条短测试消息到目标群。
- 邮件：先向目标邮箱发送一封短测试邮件。
- GitHub：先确认远端、分支、权限和认证状态。

如果失败，要区分代理问题、沙箱网络限制、权限不足、凭据错误和目标对象错误。不要把“本地配置存在”当作“通道可用”。

### 2. 再确认信息披露边界

凡是要把本地工作区内容发到外部系统，都应先明确告诉用户会外发哪些信息，并获得确认。

需要特别确认的内容包括：

- 工作区路径、项目名、分支名和远端地址。
- `AGENTS.md`、issue log、skills 清单等治理文件摘要。
- 错误复盘、内部规则缺口、自动化配置和凭据状态。
- 任何来自本地代码、文档或私有对话的内容。

技术链路验证通过，不代表可以外发内容。外部投递前仍要确认披露风险。

### 3. 最后验证客户端展示效果

API 返回成功只代表消息被接收，不代表用户实际看得到完整内容。长报告尤其要验证目标客户端的展示行为。

常见问题：

- Markdown 被目标系统转换成卡片，只显示标题。
- 多行文本在聊天流里只展示第一行。
- 长文本被折叠，用户需要额外点击才可读。
- 附件上传成功，但目标群权限不足或文件类型不可预览。

当报告较长时，优先使用“短摘要 + 附件”的模式。

## 推荐投递模式

### 短摘要

适合直接发在聊天流中：

```text
Codex 周报｜发现 3 类复发风险：skill 清单遗漏、外部通道验证不足、Windows 命令预检缺失。完整报告见附件。
```

### 完整报告

适合写入文件后作为附件发送：

```text
<WORKSPACE_ROOT>\temp\codex-error-audit-report-YYYY-MM-DD.md
```

推荐格式：

- `.md`：适合保留标题、列表和代码片段。
- `.txt`：适合最大兼容性和直接预览。

不要把完整报告塞进单条聊天消息。聊天消息只承载摘要和行动提醒，报告正文用附件承载。

## 自动化提示词要求

自动化说明应包含：

- 读取文件和工作目录。
- 运行时间和时区。
- 输出文件路径。
- 外部发送命令或通道。
- 重复复盘识别和合并建议规则。
- 报告正文不得直接塞进聊天消息。
- 失败时要报告失败原因和下一步建议。

示例：

```text
读取 <WORKSPACE_ROOT>\codex-issue-log.md 和 <WORKSPACE_ROOT>\AGENTS.md，生成完整审计报告。
识别 codex-issue-log.md 中是否存在重复复盘或同一问题的多次记录；不要在每轮对话中检查旧问题，只在本定时任务中集中整理。
如发现重复问题，输出建议合并到哪一条、建议保留哪些发生记录、建议如何标注已解决/复发/观察中。
将完整报告写入 <WORKSPACE_ROOT>\temp\codex-error-audit-report-YYYY-MM-DD.md。
向外部群聊只发送一条单行摘要，并把完整报告作为附件发送。
不要自动修改治理文件；只有用户明确授权后，才合并或改写 codex-issue-log.md。如发送失败，说明失败原因和下一步处理建议。
```

## 复盘要求

以下情况应记录到 `codex-issue-log.md`：

- 自动化创建后没有真实验证投递链路。
- 外部发送被披露风险审核拦截。
- API 返回成功但客户端展示不符合预期。
- 报告格式、附件、权限或目标群配置导致返工。
- 自动化修改了不该修改的文件，或没有修改应维护的治理文件。
